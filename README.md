# Gurukul

A self-evolving documentary of LLM research, generated by a Teacher /
Student pair of Databricks-served foundation models, with per-topic
confidence tracking, open-problems capture, and a path from study
journal to research proposal outline.

The public site is a static Docusaurus build. There is no live model
endpoint on the deployed site — your journey is generated on your own
machine and committed as MDX. That keeps the published site cheap to
host and immune to runtime hallucination drift.

## What you get

- **Left sidebar** = the journey, ordered. Topics build progressively;
  comparison chapters of frontier models (GPT, Claude, Kimi, Qwen,
  DeepSeek) come near the end.
- **Right side** = the topic itself, with interactive widgets:
  - `Summary` — TL;DR + bottom line.
  - `KeyAspect` — bite-sized concept cards.
  - `Gist` — code/math snippet with annotation.
  - `OpenProblems` — star the ones you want to chase.
  - `References` — every citation starts *unverified*; you cycle through
    `unverified → verified → wrong → missing`.
  - `Experiment` — checklist + free-text notes, persisted.
  - `Critique` — your pushback on the FM-generated content.
  - `ConfidenceTracker` — honest 1–5 self-rating ("could you explain
    this without notes?").
  - `ResearchSeed` — half-formed ideas you want to develop.
  - `ModelComparison` — first-class table with explicit per-cell
    confidence; `unknown` is preferred to a confident guess.
- **/journey-dashboard** — aggregated view across all topics:
  confidence map, starred problems, seeds, pushback notes, reference
  verification ratio.
- **/proposal** — picks seeds + pushback + starred problems and exports
  a JSON payload you pipe into `npm run outline` to draft a
  NeurIPS-style proposal outline.

## How content is produced

The CLI (`npm run journey`) implements an **outline → write → critique →
revise** loop by default:

1. **Teacher** outlines the chapter (3–5 sections, or comparison-table
   rows for a comparison chapter).
2. **Student** writes the chapter as strict JSON matching a fixed
   schema.
3. **Teacher** critiques the draft; if there are major issues, the
   Student revises.
4. A deterministic Markdown renderer turns the JSON into MDX. The model
   never emits raw JSX, so the build cannot break on stray braces.
5. The new topic is appended to `docs/_topics.json` and a fresh
   `sidebars.generated.ts` is written.

Two alternative protocols ship out of the box: `socratic` (every key
aspect opens with a question) and `debate` (pro and con drafts get
synthesised, with a forced "where the field actually disagrees" aspect).

## Setup

You need:

- Node 20+
- The Databricks CLI on `PATH` (`brew install databricks` or `pip install databricks-cli`)
- A service principal in your Databricks account with `CAN_QUERY` on the
  two serving endpoints you plan to use

One-time auth setup on this machine — **no secrets in `.env`**:

```bash
databricks auth login --host https://<your-workspace>.cloud.databricks.com
# pick "OAuth (machine-to-machine)" and bind the SP. Credentials live
# in your OS keyring, never on disk in this repo.

# verify
databricks auth token --host https://<your-workspace>.cloud.databricks.com
```

Then configure the project:

```bash
cp .env.example .env
# fill in DATABRICKS_HOST and the two endpoint names. Optionally set
# DATABRICKS_CONFIG_PROFILE if you maintain multiple profiles.
npm install
```

Smoke-test the whole pipeline without touching Databricks at all:

```bash
JOURNEY_DRY_RUN=1 npm run journey
```

Generate your first real topic:

```bash
npm run journey
```

At runtime the journey CLI shells out to `databricks auth token` and
caches the access token in-process. The token's `expiry` is honored;
on a 401 from the serving endpoint, the cache is dropped and a fresh
token is minted automatically.

Preview locally:

```bash
npm start
```

## Caveats — read these before you cite anything

- **No web search, no retrieval.** The FM speaks from its training data.
  Specific facts (parameter counts, training-token counts, RL recipes,
  ablation numbers) will be wrong some of the time. That's why every
  `Reference` starts `unverified` — verify each one before you cite it.
- **The model-comparison chapters are explicitly qualitative.** We
  compare *how each model is built and how the labs differentiate*, not
  exact metrics. Each cell carries a confidence; `unknown` is correct
  when there is no public info — better than a confident guess.
- **The proposal outline is an outline.** The experiments are still on
  you. NeurIPS reviewers will not accept a model-generated outline as a
  paper.
- **The "confidence tracker" is a self-rating, not a learning score.**
  It measures how confident you feel, not what you know.

## Deploy

The included GitHub Actions workflow (`.github/workflows/deploy.yml`)
builds and publishes to GitHub Pages on every push to `main`. Set these
repo settings:

- **Settings → Pages → Source:** GitHub Actions.
- Optional repository secrets:
  - `SITE_URL` (e.g. `https://your-username.github.io`)
  - `SITE_BASE_URL` (e.g. `/gurukul/`)
